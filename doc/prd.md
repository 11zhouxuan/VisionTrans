# 📝 产品需求文档 (PRD)：AI 视觉划词翻译工具 (Desktop MVP)

## 一、 产品概述

* **产品代号**：VisionTrans (暂定名)
* **核心价值**：打破传统"选中文本->提取->翻译"的限制。通过"全局快捷键一键截屏 + 自由涂抹 + 多模态 LLM 视觉识别"，实现对屏幕上任意元素（图片、加密文档、视频字幕、复杂排版）的精准、极速翻译。
* **支持平台**：macOS 11+ / Windows 10+ (64-bit)
* **当前版本**：v1.0 MVP

## 二、 目标用户与痛点

* **目标人群**：经常阅读外文资料的科研人员、设计师、海外游戏玩家、视频消费者。
* **解决的痛点**：
  1. **无法选中的痛点**：图片、视频字幕、PDF 加密文档中的外文无法用传统划词工具选中。
  2. **上下文割裂的痛点**：传统翻译工具只翻译纯文本，丢失了排版、语境和配图信息，导致机翻味重。
  3. **操作繁琐的痛点**：传统的"截图->打开翻译软件->上传图片->获取结果"流程太长，打断工作流。

## 三、 竞品分析

| 竞品 | 平台 | 核心方式 | VisionTrans 的差异化优势 |
| --- | --- | --- | --- |
| **Bob** | macOS | OCR + 翻译 API 两步走 | VisionTrans 使用多模态 LLM 直接理解图像上下文，翻译质量更高，无需 OCR 中间步骤 |
| **Pot** | 跨平台 | OCR + 翻译 API 两步走 | VisionTrans 的涂抹交互更自然，且 LLM 能理解排版/配图语境 |
| **Umi-OCR** | Windows | 本地 OCR 引擎 | VisionTrans 不依赖 OCR 准确率，对复杂排版、艺术字体、手写体等场景表现更好 |
| **传统划词翻译** | 各平台 | 选中文本 + 翻译 | VisionTrans 突破"必须能选中文本"的限制，覆盖图片/视频/加密文档等场景 |

**核心差异**：传统工具走的是 "OCR 识别文字 → 翻译文字" 的两步路径，OCR 错误会传导到翻译结果。VisionTrans 走的是 "多模态 LLM 直接看图理解 → 输出翻译" 的一步路径，能利用图像中的排版、配图、上下文信息给出更准确、更自然的翻译。

## 四、 核心用户旅程 (User Journey)

这是用户使用产品的标准流程，主打**"用完即走，不打断沉浸感"**：

1. **唤醒**：用户在任何软件中按下全局快捷键（默认 `Alt+Q` 或 `Option+Q`）。
2. **冻结**：屏幕瞬间被"冻结"（底层逻辑：截取当前全屏并铺满在一个无边框透明窗口上），鼠标指针变为"荧光笔"样式。
3. **涂抹**：用户用鼠标在屏幕上涂抹或框选需要翻译的外文区域（留下半透明的高亮轨迹）。
4. **松手即翻**：用户松开鼠标左键，高亮轨迹消失，屏幕立刻恢复可操作状态（解除冻结）。
5. **展示**：1~2秒后，在刚刚涂抹的位置附近，弹出一个小巧、精致的无边框卡片，展示 LLM 结合图像上下文给出的翻译结果。
6. **关闭**：用户点击屏幕其他任意区域，或按下 `Esc` 键，翻译卡片消失。

## 六、 功能需求拆解 (Functional Requirements)

### 1. 全局控制模块

* **快捷键监听**：支持在系统后台常驻，监听用户自定义的全局快捷键。
* **单例模式**：确保程序在后台只能运行一个实例。
* **系统托盘**：
  * 托盘图标常驻系统通知区域
  * 右键菜单包含：设置、暂停/恢复监听、关于、退出
  * 左键单击托盘图标：显示/隐藏设置窗口

### 2. 屏幕捕获模块 (Capture & Freeze)

* **极速截屏**：按下快捷键后，毫秒级获取当前屏幕像素数据（需支持多显示器场景，截取当前鼠标所在的屏幕）。
* **全屏遮罩**：创建一个置顶的、全屏无边框窗口，将截图设为背景，做到"视觉无缝冻结"。
* **HiDPI / 多分辨率适配**：
  * 正确处理 Retina (2x) 及其他高分辨率屏幕的缩放比例
  * 多显示器场景下，不同屏幕可能有不同的缩放比例，需分别处理
  * 截图的物理像素与逻辑像素映射需正确，确保遮罩层与实际屏幕内容完美对齐

### 3. 交互画布模块 (The Canvas)

| 功能项 | 描述 |
| --- | --- |
| **涂抹高亮** | 鼠标按下并拖动时，绘制带有一定宽度的半透明线条（类似马克笔）。 |
| **矩形框选** | 支持传统的拖拽画框方式（作为涂抹的补充）。 |
| **取消操作** | 右键点击画布，或按下 `Esc`，退出本次截图状态。 |
| **图像裁剪** | 鼠标松开后，根据涂抹轨迹生成**最小外接矩形**图像（MVP 方案），转换为 Base64 格式发送给 LLM。 |

> 💡 **MVP 决策**：图像裁剪策略 MVP 阶段采用"最小外接矩形"方案（实现简单、效果可靠）。后续版本可优化为"非涂抹区域模糊/黑白处理"以保留更多上下文信息。

### 4. AI 翻译引擎模块

* **API 设置**：允许用户在设置页面输入自己的大模型 API Key（支持 OpenAI GPT-4o, Claude 3.5 Sonnet, Gemini Pro Vision 等兼容 OpenAI 格式的接口）。
* **自定义 API Endpoint**：支持用户配置自定义 API 地址（方便使用代理或第三方中转服务，解决国内用户网络访问问题）。
* **Prompt 组装**：将处理后的图片配合系统预设的 Prompt 发送给 LLM。
  * *预设 Prompt 示例*："识别图片中高亮部分的文本，结合整体图像上下文，翻译为{目标语言}。直接输出翻译结果，保持原意和专业词汇的准确性。"
* **目标语言**：MVP 支持中文（默认）和英文两种目标语言，源语言由 LLM 自动识别。
* **加载状态**：请求期间，在屏幕特定位置显示轻量的 Loading 动画。

### 5. 结果展示模块

* **悬浮卡片**：无边框、带阴影的圆角卡片，不抢夺当前系统焦点（`NO_ACTIVATE`）。
* **智能定位**：卡片默认出现在用户松开鼠标的坐标附近，自动避开屏幕边缘。
* **卡片内容**：
  * 翻译结果文本（主体）
  * 源语言标识（如 "EN → 中文"）
  * 一键复制按钮
  * 重新翻译按钮（用于结果不满意时重试）
* **卡片交互**：卡片可鼠标拖拽移动位置。
* **关闭方式**：点击屏幕其他区域 / 按 `Esc` / 点击卡片关闭按钮。

### 6. 首次启动引导模块 (Onboarding)

首次启动时，展示一个简洁的引导流程：

| 步骤 | 内容 | 说明 |
| --- | --- | --- |
| Step 1 | 欢迎页 | 产品简介 + 核心功能演示动画 |

| Step 3 | API Key 配置 | 引导用户输入 LLM API Key，提供获取 Key 的链接，支持测试连接 |
| Step 4 | 快捷键确认 | 展示默认快捷键，允许用户修改 |
| Step 5 | 完成 | 引导用户尝试第一次翻译 |

## 七、 异常处理与降级策略

| 异常场景 | 处理方式 |
| --- | --- |
| **API Key 未配置** | 首次涂抹翻译时，弹出提示引导用户前往设置页面配置 API Key |
| **API Key 无效 / 认证失败** | 卡片中显示错误提示："API Key 无效，请检查设置"，附带"前往设置"按钮 |
| **网络超时（>10s）** | 卡片中显示超时提示 + "重试"按钮，建议用户检查网络或配置代理 |
| **网络不可用** | 卡片中显示离线提示："当前无网络连接，请检查网络设置" |
| **LLM 返回异常 / 空结果** | 卡片中显示："未能识别翻译内容，请尝试重新涂抹更大的区域" |
| **涂抹区域无文字** | LLM 会返回相应提示，卡片中显示："未检测到需要翻译的文本" |
| **涂抹区域过小（<20px）** | 忽略本次操作，不发送请求 |
| **macOS 屏幕录制权限未授予** | 截屏失败时，弹出系统级提示引导用户授权，并打开系统设置对应页面 |
| **API 调用频率限制 (Rate Limit)** | 卡片中显示："请求过于频繁，请稍后再试" |

## 八、 非功能性需求 (Non-Functional Requirements)

### 性能指标

| 指标 | 目标值 | 说明 |
| --- | --- | --- |
| 快捷键到屏幕冻结 | ≤ 200ms | 用户感知"即时响应" |
| 涂抹到结果展示 | ≤ 3s | 取决于 LLM API 响应速度，本地处理 ≤ 500ms |
| 后台常驻内存占用 | ≤ 30MB | 非活跃状态下 |
| 安装包体积 | ≤ 15MB | 不含系统 WebView（Windows 需 WebView2 Runtime，通常已预装） |

### 其他要求

* **跨平台兼容**：
  * Windows: 支持 Win10/Win11，无需特殊权限。若系统未安装 WebView2 Runtime，需在安装时自动引导安装。
  * macOS: 首次启动需引导用户在系统设置中授予"屏幕录制 (Screen Recording)"权限。
* **隐私安全**：截图数据仅存在于本地内存中，用完即焚，不写磁盘。仅在用户涂抹后，将局部图片通过 HTTPS 加密发送给 LLM 服务。

## 九、 成本预估与用户须知

由于每次翻译需要向 LLM 发送图片，Token 消耗相对较高，用户需自行承担 API 调用费用：

| 模型 | 单次翻译预估费用 | 说明 |
| --- | --- | --- |
| GPT-4o | ~$0.005 - $0.02 | 取决于图片大小和分辨率 |
| Claude 3.5 Sonnet | ~$0.005 - $0.015 | 取决于图片大小和分辨率 |
| Gemini Pro Vision | ~$0.001 - $0.005 | 相对更经济 |

> 💡 **建议**：在设置页面中增加"费用提示"说明，让用户了解每次翻译的大致成本。后续版本可考虑增加本地 OCR 降级方案，在简单场景下减少 API 调用。

```
│                    用户操作层                              │

│  │ (涂抹/框选)  │  │ (翻译展示)   │  │  (配置管理)   │   │
│  └──────┬──────┘  └──────┬───────┘  └───────────────┘   │
│         │                │                               │
└─────────┼────────────────┼───────────────────────────────┘
          │                │
┌─────────▼────────────────▼───────────────────────────────┐
│                  Tauri 后端 (Rust)                         │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────┐  │
│  │ 全局快捷键    │  │ 屏幕截图      │  │ 窗口管理       │  │
│  │ 注册/监听     │  │ (多屏/HiDPI) │  │ (遮罩/卡片)    │  │
│  └──────────────┘  └──────────────┘  └────────────────┘  │

└──────────────────────────┬───────────────────────────────┘
                           │ HTTPS
┌──────────────────────────▼───────────────────────────────┐
│                  外部 LLM API 服务                         │
│         OpenAI / Claude / Gemini (兼容 OpenAI 格式)       │
└──────────────────────────────────────────────────────────┘
```

### 数据流

1. **快捷键触发** → Rust 层捕获全局快捷键事件
2. **屏幕截图** → Rust 调用系统 API 截取当前屏幕（处理 HiDPI 缩放）
3. **创建遮罩** → Rust 创建全屏无边框窗口，前端将截图渲染为 Canvas 背景
4. **用户涂抹** → 前端 Canvas 记录涂抹轨迹坐标
5. **图像裁剪** → 前端根据涂抹轨迹计算最小外接矩形，裁剪 Canvas 图像并转为 Base64
6. **API 调用** → 前端/Rust 层将 Base64 图片 + Prompt 发送给 LLM API
7. **结果展示** → 前端接收翻译结果，在涂抹位置附近渲染悬浮卡片

### 下一步行动建议

PRD 已经理清了产品的骨架。对于这个桌面 MVP，最难攻克的技术核心是：**如何用 Rust 在按下快捷键的瞬间截取屏幕，并把它完美贴合在一个 Tauri 无边框窗口的 Canvas 上。**

建议的开发优先级：

1. **P0 - 核心链路**：全局快捷键 → 截屏 → 冻结遮罩 → 涂抹 → 裁剪图片 → 调用 LLM → 展示结果
2. **P1 - 基础配置**：设置页面 + 系统托盘 + 单例模式
3. **P2 - 体验优化**：首次启动引导 + 异常处理 + HiDPI 适配
